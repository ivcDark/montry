# Редизайн Montry

Дата обновления: 2026-06-07.

Цель редизайна - сделать Montry компактным, понятным и удобным SaaS-инструментом для владельцев небольших бизнесов, фрилансеров, SEO-специалистов и веб-студий. Интерфейс должен конкурировать по удобству с `prufen.ru`, `enterno.io`, `statuser.cloud` и `monisite.ru`, но оставаться самостоятельным продуктом.

## Продуктовый фокус

- Пользователь должен быстро добавить сайт или домен и сразу получить базовые проверки.
- Управление мониторингами должно быть проще, чем чтение документации: включить, выключить, изменить интервал, добавить SSL/DNS/robots/API/порт.
- Billing должен быть прозрачным: базовый тариф + подключенные платные опции + итоговая сумма.
- Сервис не должен выглядеть размашистым. Это рабочий кабинет, а не промо-лендинг.

## Визуальное направление

- Основная палитра: светлая, чистая, с зелеными акцентами в духе `prufen.ru`.
- Зеленый использовать для основного CTA, успешных статусов, активных переключателей и выбранных тарифов.
- Фон интерфейса: светлый нейтральный, без тяжелых темных панелей.
- Карточки и панели: компактные, с небольшим радиусом, умеренной тенью или тонкой границей.
- Типографика: спокойная, читаемая, без гигантских заголовков в dashboard-части.
- В таблицах и списках важнее сканируемость, чем декоративность.

Запрещено для кабинета:

- большие hero-блоки;
- огромные карточки с малым количеством информации;
- декоративные градиентные пятна и фоны;
- интерфейс, где один сайт занимает половину экрана без причины;
- маркетинговая подача внутри рабочих экранов.

## Основные экраны

### Dashboard

Компактный обзор:

- общее состояние: `все хорошо`, `есть проблемы`, `проверяется`;
- количество сайтов;
- активные инциденты;
- ближайшие SSL/domain expiration warnings;
- последние проверки;
- быстрые действия: добавить сайт, проверить сейчас, настроить уведомления.

### Сайты и ресурсы

Список должен поддерживать:

- поиск;
- фильтр по проекту/клиенту;
- фильтр по статусу;
- быстрые badges активных мониторингов;
- быстрые действия без перехода в глубокую форму: pause/resume, check now, настройки.

Карточка сайта должна показывать:

- URL/домен;
- статус доступности;
- SSL;
- domain expiration;
- DNS;
- robots.txt;
- paid monitors: sitemap.xml, TCP ports, API endpoints;
- историю последних проверок;
- открытые инциденты;
- кнопки настройки каждого мониторинга.

### Настройка мониторингов

Пользователь должен видеть типы мониторинга как понятный список переключателей/секций:

- базовые проверки включаются простыми toggles;
- у каждой проверки есть компактная настройка: interval, timeout, expected values;
- платные проверки показывают цену и единицу тарификации прямо в настройке;
- при включении платной проверки UI пересчитывает будущую стоимость тарифа;
- для TCP ports и API endpoints должен быть удобный повторяемый список с добавлением строк.

### Billing

Страница тарифа должна быть конфигуратором, а не просто тремя карточками:

- текущий тариф и его использование;
- Free/Pro/Plus как компактное сравнение;
- блок платных опций;
- расчет итоговой стоимости;
- предупреждения о лимитах;
- понятные empty/error states.

Для веб-студий важно быстро понять: сколько клиентов/сайтов помещается в тариф, сколько будет стоить добавить еще 5 сайтов, какие проверки стоят отдельно.

### Уведомления

Настройка каналов должна быть рядом с рабочими сценариями:

- email доступен всем;
- Telegram доступен в Pro и Plus;
- при недоступном канале показывать причину и CTA смены тарифа;
- после сохранения показывать toast.

## Toast сообщения

Все действия пользователя должны сопровождаться аккуратными toast:

- success: сайт добавлен, мониторинг сохранен, тариф обновлен, проверка запущена;
- warning: лимит почти исчерпан, проверка пока работает как заглушка, Telegram недоступен на Free;
- error: ошибка сохранения, не удалось запустить проверку, платеж не подтвержден;
- info: проверка поставлена в очередь, результат появится после poller callback.

Toast должен быть коротким, с понятным текстом и, где полезно, с действием: `Открыть настройки`, `Посмотреть тариф`, `Повторить`.

## Конкурентные ориентиры

Использовать конкурентов как UX-ориентиры:

- `prufen.ru` - светло-зеленая визуальная система, простая подача проверок, понятный CTA.
- `enterno.io` - удобная группировка инструментов и быстрый доступ к проверкам.
- `statuser.cloud` - акцент на мониторинг, уведомления и состояние сервисов.
- `monisite.ru` - понятная категория сервиса для мониторинга сайта.

Предложения по улучшению Montry приветствуются в следующих итерациях. Если у конкурентов сценарий сделан заметно удобнее, нужно предлагать адаптацию под Montry, особенно для тарифов, добавления мониторингов и настройки уведомлений.

## Реализация новых типов

Новые типы мониторинга должны появляться в интерфейсе до полной реализации Go poller:

- `dns`;
- `robots_txt`;
- `sitemap_xml`;
- `tcp_port`;
- `api_endpoint`.

Для каждого типа нужен definition в Laravel `CheckTypes`, форма настроек, сохранение monitor и понятный UI-статус. Go poller обработку можно реализовать позже. До этого результаты должны быть заглушками и не должны создавать incidents.
